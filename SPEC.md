# まちのできごと — サービス仕様書

> Version 0.1 / 2026-04-28 起草
> ステータス：**ドラフト**（既存プロトタイプを下敷きに、これから合意形成していく仕様）

このドキュメントは、地域のイベント情報を投稿・告知・チケット販売できるサービス「まちのできごと」の仕様書です。すでに動いている [プロトタイプ](_extracted/index.html) を出発点として、本番化に向けて何を作るか／何を後回しにするかを定義します。

---

## 1. プロジェクトの目的

### 1.1 ひとことで

> **近所で誰かが開いているイベントを、もっと簡単に見つけられる場所をつくる。**

### 1.2 解きたい課題

地域のイベントは、こども食堂、認知症カフェ、朝市、読書会、まちのピアノ、福祉系の集まり、個人カフェの月一会など、**主催者の規模が小さく分散している**のが特徴です。情報は次のような場所にバラバラに置かれています：

- 主催者の手書きチラシ（公民館・道の駅）
- LINE オープンチャット / Instagram
- 自治体の広報誌
- 商店街・回覧板

結果として、**「行きたい人」と「開いている人」が出会いにくい**。Peatix・Connpass のような既存サービスは IT 勉強会や都市部のイベント向けで、トーンも操作感も合いません。

### 1.3 提供価値

1. **参加者向け**：「自分のまちで、近い未来に何があるか」を一目で見つけられる。地図・カテゴリ・お気に入り主催者から探せる。
2. **主催者向け**：チラシを書く感覚でイベントを投稿でき、参加登録・チケット販売・参加者への連絡・終了後のレポート公開まで、一つの場所で完結する。
3. **シリーズの記録**：認知症カフェ第十八回、スープのタイム〈五月〉のように **続いている集まり** を「シリーズ」としてまとめ、過去の積み重ね（開催回数・延べ参加者数・写真入りレポート）を残していく。

### 1.4 当初のターゲット地域

- ギフト県のみ市 / きせ市 / うじょぐ市 / ギフト市 / にか市 / みじた市 / かみはら市
- 拡大方針はサービスのトーン（和紙・町家・ゆっくりした文体）が馴染む地域から。

## 2. 想定ユーザー

### 2.1 ペルソナ

#### 参加者ペルソナ A：地域に関心のある主婦・退職世代（50–70代）

- スマホは LINE と地図アプリ、Yahoo! 天気が中心。
- 「うちの近所、最近何かやってないかしら」という軽い動機で覗く。
- アプリストアからのインストールはハードル高め → **PWA（ブラウザから保存）** が望ましい。
- 漢字が読めて画面文字が大きいことを評価する。

#### 参加者ペルソナ B：移住者・二地域居住の働く世代（30–40代）

- iOS / Android、SNS から流入。
- 「土日に子どもを連れて行ける場所」「夜にひとりで行ける場所」を探す。
- カテゴリ（読書、音楽、こども食堂、福祉）でのフィルタを使う。

#### 主催者ペルソナ C：地域包括支援センター・NPO 等の小組織

- 月1回〜の定例イベントを長年続けている（例：認知症カフェ）。
- IT は得意でないが、Excel と Word は使える。
- **参加者の名簿管理と、終了後の報告書づくり** が手間。それをシステム側で吸収できると価値が大きい。

#### 主催者ペルソナ D：個人カフェ・読書会主催者

- 月数回、有料のスープ会・読書会を開く（参加費 500〜1,500円）。
- 参加費は当日現金 or PayPay、もしくは **事前決済できると当日が楽**。
- レポートを写真付きで残せると、シリーズの「過去の積み重ね」がそのまま集客資産になる。

### 2.2 役割（ロール）

| ロール | 概要 | 主な権限 |
| --- | --- | --- |
| **ゲスト** | 未ログイン | イベント閲覧・地図閲覧・無料 RSVP（メール認証ベース） |
| **参加者** | ログイン済み一般ユーザー | お気に入り、購入履歴、チケット表示、おしらせ受信、参加者プロフィール |
| **主催者** | イベントを投稿できるユーザー | イベント・シリーズの作成/編集、参加者一覧、レポート投稿、チケット種別設定、売上振込 |
| **シリーズ管理者** | シリーズ単位での共同管理者 | そのシリーズに紐づく全イベントの編集・参加者管理・レポート投稿 |
| **運営（まちのできごと）** | サービス運営者 | 主催者承認、不適切投稿の対応、地域マスタ管理、特集編集 |

ロールはユーザーに対する付与で、1ユーザーが「参加者」かつ「主催者」を兼ねるのが標準（参加してる側が次に主催者になる流れを後押ししたい）。

## 3. ユースケースと体験ストーリー

### 3.1 体験ストーリー：参加者がイベントを見つけて参加する

1. 高田さんが LINE のともだちから「いいのあるよ」と URL をもらう。
2. ブラウザで開いたら、トップに「あなたのまちで、きょう何かが始まる」と書かれている。
3. ホーム画面に追加（PWA）し、その日から主催者をひとり「♥ お気に入り」にする。
4. 翌週、お気に入り主催者が新しい会を開いたら **おしらせ通知** が届く。
5. ボタンひとつで参加登録（または有料チケット購入）。確認メールに **当日見せるチケット QR** が添付される。
6. 当日、主催者がスマホで QR を読んで受付。
7. 翌週、主催者が **開催レポート** に写真を載せて公開。高田さんはその通知を受け取り、次回も行きたくなる。

### 3.2 体験ストーリー：主催者が初めて投稿する

1. 一般ユーザーとして使っていた人が、ダッシュボードから「主催者になる」を申請。
2. 名前・連絡先・主催する活動の説明を入力（無料イベントだけならこれで OK）。有料チケットを発行する場合は Stripe Connect の本人確認に進む。
3. 「+ 新規イベント」から、**チラシを書く感覚でフォームに記入**。タイトル / 日時 / 会場 / 参加費 / 説明 / 画像。
4. シリーズに紐づけるか選択（既存の「認知症カフェ in のみ」に第十九回として追加、など）。
5. プレビュー → 公開。即時に検索結果と地図に反映。
6. お気に入り登録している人や、過去にこのシリーズに参加した人へ **おしらせ通知**（メール / Push）が飛ぶ。

## 4. 機能要件

各機能は **MVP（必須）** / **Phase 2（重要）** / **Phase 3（後回し可）** に分類します。MVP の定義は §11 を参照。

### 4.1 イベントを発見する（探す）

| | 内容 | フェーズ |
| --- | --- | --- |
| F-DSC-1 | 地域（市区町村）でフィルタ。複数選択可。 | MVP |
| F-DSC-2 | カテゴリタグでフィルタ。階層型（集まり・交流 / 食 / 学び / 文化 / まち / スポーツ / ビジネス / 福祉 …）。 | MVP |
| F-DSC-3 | 日付レンジでフィルタ（今週末、今月、月選択）。 | MVP |
| F-DSC-4 | テキスト検索（タイトル・説明・主催者名・会場）。 | MVP |
| F-DSC-5 | 地図ビュー。ピンタップで詳細プレビュー、リスト連動。 | MVP（簡易）/ Phase 2（実地図 API） |
| F-DSC-6 | お気に入り：イベント / シリーズ / 主催者の3種。 | MVP |
| F-DSC-7 | 「最近開催されました」レポート一覧。 | MVP |
| F-DSC-8 | おすすめ・似ているイベントの推薦（ルールベースで OK）。 | Phase 2 |
| F-DSC-9 | 共有用 OGP 画像の自動生成（イベントごと）。 | Phase 2 |

### 4.2 イベントを投稿する

| | 内容 | フェーズ |
| --- | --- | --- |
| F-PST-1 | 新規イベントフォーム（タイトル・サブタイトル・日時・会場・住所・定員・参加費・説明・タグ）。 | MVP |
| F-PST-2 | シリーズへの紐付け。「新しいシリーズを作る」も同フローで可能。 | MVP |
| F-PST-3 | 開催形式：対面 / オンライン / ハイブリッド。オンラインの場合は URL と接続方法案内を保持。 | MVP |
| F-PST-4 | チラシ画像のアップロード（最大数枚）。なければテキストから自動でチラシ風レイアウトに組版。 | MVP（自動組版は簡易版）/ Phase 3（高度な組版） |
| F-PST-5 | 下書き保存と公開。公開済みでも編集可能（ただし価格・定員・日時の変更は購入者へ通知）。 | MVP |
| F-PST-6 | 過去イベントの複製で次回を起票（「第十八回 → 第十九回」）。 | MVP |
| F-PST-7 | 公開範囲：誰でも / 主催者のフォロワーのみ / 限定 URL のみ。 | Phase 2 |
| F-PST-8 | 共催者・シリーズ管理者の招待。 | Phase 2 |

### 4.3 参加登録 / チケット販売

「**まちのできごと**」の根幹のひとつ。MVP では無料登録、Phase 2 で有料チケット。

#### 4.3.1 無料 RSVP（MVP）

| | 内容 |
| --- | --- |
| F-RSV-1 | 3ステップフォーム：基本情報 → 参加について → 確認。プロトタイプ準拠。 |
| F-RSV-2 | 必須項目：氏名・メール。任意：ふりがな・電話・参加頻度・人数・備考・配慮事項。 |
| F-RSV-3 | 完了画面で **登録番号** を発行（例：`#MC-1234`）。確認メールに同番号と当日案内。 |
| F-RSV-4 | 定員管理。残席表示。満席時はキャンセル待ちに自動振替。 |
| F-RSV-5 | キャンセル：参加者がマイページから前日まで可能。主催者は任意の締切を設定可。 |
| F-RSV-6 | 参加者の同意取得：個人情報の利用目的に明示同意（チェックボックス必須）。 |

#### 4.3.2 有料チケット販売（Phase 2）

| | 内容 |
| --- | --- |
| F-TKT-1 | チケット種別を複数設定可。例：一般 1,000 / 学生 500 / こども無料 / ペア 1,800。 |
| F-TKT-2 | 在庫管理（種別ごと、全体）、販売期間、購入上限（1人◯枚まで）。 |
| F-TKT-3 | 決済は Stripe Connect Standard。**主催者が直接受け取る**形（プラットフォーム手数料を差し引き）。 |
| F-TKT-4 | 支払い方法：カード（Stripe）。日本での利用前提でコンビニ・銀行振込・PayPay は Phase 3。 |
| F-TKT-5 | チケット保有者には **QR コード入りデジタルチケット** をメール + マイページで発行。 |
| F-TKT-6 | 主催者は当日、専用受付画面（Web）でカメラ越しに QR を読み取り、チェックイン状態を更新。オフラインでも一時動作（後同期）。 |
| F-TKT-7 | 払戻し方針：開催◯日前まで全額、それ以降◯%、当日不可、を主催者がイベントごとに設定。 |
| F-TKT-8 | 当日券（オンライン販売をしながら、主催者が手動で当日購入を発行）。 |
| F-TKT-9 | 領収書 PDF 発行（参加者がマイページから DL）。 |
| F-TKT-10 | プラットフォーム手数料：仮置き **販売額の 5% + Stripe 手数料**（要決定）。 |

#### 4.3.3 ハンドル不可決定事項（要協議）

- 手数料率の最終決定
- インボイス（適格請求書）対応の必要性 → 主催者が個人/法人どちらを取るかで変わる
- 「無料イベントは主催者から手数料を取らない」かどうか

### 4.4 おしらせ・通知

「投稿してお知らせする」ためのコア機能。

| | 内容 | フェーズ |
| --- | --- | --- |
| F-NOT-1 | **お気に入り主催者**が新規イベントを公開したとき、フォロワーに通知。 | MVP |
| F-NOT-2 | **シリーズ参加経験者**に次回開催を通知（オプトアウト可）。 | MVP |
| F-NOT-3 | **登録済みイベント**のリマインド通知（前日・当日朝・直前1時間）。 | MVP |
| F-NOT-4 | 主催者から参加者全員へのお知らせ送信（直前案内、雨天中止、持ち物追加等）。 | MVP |
| F-NOT-5 | 通知チャネル：**メール**（必須・MVP）、**Web Push**（PWA・Phase 2）、**LINE 公式アカウント連携**（Phase 2）。 | MVP / Phase 2 |
| F-NOT-6 | 通知設定：チャネルごとの ON/OFF、配信時間帯指定、ダイジェスト（週1回まとめ）。 | Phase 2 |
| F-NOT-7 | システムからのお知らせ（運営からのお知らせ、利用規約変更）。 | MVP |

実装メモ：MVP ではメール (Resend / SendGrid 等) と、PWA でブラウザ Push（VAPID）を最低限。LINE は LINE Messaging API でユーザーが連携した場合に追加配信。

### 4.5 シリーズ

| --- | --- | --- |

### 4.6 開催レポート

| | 内容 | フェーズ |
| --- | --- | --- |
| F-RPT-1 | 主催者・シリーズ管理者がレポートを投稿（写真最大8枚 + 本文 + 投稿者名）。 | MVP |
| F-RPT-2 | 1イベントに **複数レポート可** （事務局視点・主催者視点・参加者視点など）。 | MVP |
| F-RPT-3 | レポートに **レシピ・配布物** を構造化データで添付できる。PDF 配布物（[handout.html](_extracted/handout.html)）として書き出し可。 | MVP（PDF 出力は Phase 2） |
| F-RPT-4 | レポートを「未投稿のままにしない」ためのリマインド。 | MVP |
| F-RPT-5 | 写真の顔ぼかし支援（任意）。 | Phase 3 |

## 7. 非機能要件

### 7.1 パフォーマンス

- トップの初回表示（LCP）：3G 相当で 2.5 秒以内、4G で 1.5 秒以内。
- イベント詳細：1.5 秒以内。
- 検索フィルタの操作：100 ms 以内（クライアント側で）。

### 7.2 アクセシビリティ

- WCAG 2.1 AA 準拠を目指す。
- ターゲットの一部が高齢者であるため、**フォントサイズの可変** と **コントラスト比 4.5:1 以上** を必ず満たす。
- フォーカスリング・キーボード操作完備。
- スクリーンリーダー（VoiceOver / TalkBack）で主要導線を歩けること。

### 7.3 国際化・地域化

- 第1版は **日本語のみ**。i18n フレームワークは入れておくが、英語・中国語等の追加は需要を見て。
- 日付・時刻は和暦/西暦どちらでも読める表記（`5月8日(金) 13:30 – 15:30`）。
- 通貨は日本円のみ（Phase 2 のチケットも JPY 固定）。

### 7.4 プライバシー・個人情報

- 取得する個人情報：氏名・メール・電話（任意）・住所はチケット送付に使う場合のみ。
- 利用目的を登録時に明示し、明示同意（チェックボックス）を必須にする。
- 主催者には参加者の個人情報を表示するが、CSV エクスポートにはアクセスログを取る。
- 退会時：30日以内に個人特定情報を削除（注文履歴は会計上の必要最低期間 7 年で匿名化保持）。

### 7.5 セキュリティ

- 認証：パスワードレス（メールマジックリンク + OAuth：Google / LINE）を推奨。
- 通信：すべて TLS 1.2 以上。
- データベース：行レベルセキュリティ（Supabase RLS）で、主催者は自イベントの参加者のみ閲覧可能。
- 決済情報はサービス側に保持しない（Stripe にトークン化）。
- レートリミット：登録フォーム・OTP 発行・通知送信。
- 監査ログ：主催者が参加者リストを CSV で落とした、メッセージ一斉送信した、などの操作。

### 7.6 PWA / オフライン

- マニフェスト・アイコン・Service Worker 完備。
- オフライン時：閲覧済みイベントの詳細は cache-first で読める。
- 受付（チェックイン）画面はオフラインでも一時動作 → オンライン復帰時に同期。

### 7.7 SEO・OGP

- 各イベント・シリーズ・主催者ページに固有の URL（パーマリンク）。
- イベント URL：`/event/<region>-<short-id>-<slug>` （例：`/event/mino-9k2-ninchisho-cafe-18`）。
- OGP 画像はチラシから自動生成（テキストのみのイベントは自動組版）。
- 構造化データ：`Event` schema.org をすべての公開イベントで配信。


### 9.2 ターゲット構成（Phase 2, チケット販売）

追加コンポーネント：

- **Stripe Connect Standard**：主催者ごとに Stripe アカウントを連携。プラットフォームは `application_fee_amount` で手数料を取得。
- **Webhook 受信**：`payment_intent.succeeded` 等で `Registration` を `confirmed` に更新。
- **Web Push**：VAPID + Service Worker。
- **LINE Messaging API**：通知の追加チャネル。
- **QR コード生成**：`qrcode` ライブラリ。チケットトークンは HMAC 署名付き。

### 9.3 採用判断のポイント（要相談）

- **Next.js + Supabase** が一番無難。日本国内でも事例多い。
- **Cloudflare Workers + D1** で全部やる手もあるが、PostGIS/全文検索などで Postgres があると楽。
- **モバイル**：当面 PWA のみ。ネイティブアプリは需要を見て。
- **管理画面**：Phase 1 では主催者ダッシュボードがそれを兼ねる。運営用の管理画面は Retool / Supabase Studio で代替。

## 10. 法務・運営ポリシー（要協議）

- **特定商取引法表示**：有料チケット販売開始時に必要（運営事業者・主催者それぞれ）。
- **資金決済法**：プリペイド機能（事前チャージ）は当面持たない。
- **インボイス対応**：主催者が課税事業者かどうか申告できる項目を持ち、領収書 PDF に登録番号を記載。
- **コンテンツガイドライン**：宗教勧誘・マルチ商法・成人向け・違法行為の禁止。
- **削除・通報導線**：参加者・第三者からの通報フォーム、運営の確認 → 当事者に通知 → 公開停止/削除。
- **未成年の参加**：保護者の同意取り扱い。こども食堂など多いため設計が必要。
- **キャンセル時の手数料**：主催者ポリシーに準拠。Stripe 手数料は返金しない可能性を明記。

## 11. フェーズ計画

### MVP（Phase 1）— 「投稿してお知らせする」を最小で成立させる

**ゴール**：のみ・きせのシリーズ主催者 5〜10 件に実運用で乗ってもらう。

含むもの：
- 認証（メールマジックリンク、Google）
- イベント投稿・編集・公開（無料のみ）
- シリーズ作成・運用
- 参加者の RSVP（無料）+ 確認メール + リマインド
- 開催レポート投稿
- お気に入り・主催者フォロー・新規イベント通知（メール）
- トップの探す画面・地域・カテゴリフィルタ・地図（簡易）
- マイページ（予約中・参加済・お気に入り・主催）
- 主催者ダッシュボード（概要・イベント管理・レポート投稿）
- PWA・i18n 基盤・OGP

含まないもの：
- 有料チケット販売 / Stripe
- LINE / Web Push 通知
- ハイブリッド開催の詳細制御
- アクセス解析・統計
- 共催者の招待

**目安**：開発開始から **3〜4 ヶ月**（フルタイム1〜2人換算）。

- Stripe Connect 連携・有料チケット販売・QR 受付

- 共催者の招待・シリーズ共同管理
- 売上ダッシュボード・払戻し
- アクセス解析

### Phase 3 — 「地域に根を張る」

- 自治体連携（広報誌・公民館掲示）

- ネイティブアプリ（必要なら）

## 12. オープンクエスチョン（合意が必要）

このドラフトを進めるにあたり、ユーザーと決めておきたいこと：

1. **収益モデル**：プラットフォーム手数料は何 % にするか？無料イベントから手数料を取らないか？
2. **運営主体**：このサービスを誰が運営するか（個人？法人化？）。これにより特商法表示・資金決済法・インボイスの設計が変わる。
3. **対象地域の優先順位**：のみ・きせを最初に集中するか、ギフト全域を一斉に開けるか。
4. **モバイル戦略**：当面 PWA のみで進める方針で合意できるか。
5. **データ所有権**：参加者リストの所有は主催者か、サービス共有か。
6. **既存プロトタイプの扱い**：本番リポジトリに移管して書き直すか、別リポジトリで Next.js 版を起こすか。
7. **デザインリソース**：Figma 等での画面設計を起こすか、プロトタイプの React コードを single source of truth として進めるか。
8. **画像のホスティング**：Supabase Storage / R2 / Cloudflare Images のどれにするか。
9. **チケット決済**：MVP の段階から Stripe を入れるか、本当に Phase 2 まで遅らせるか（早めに入れたほうが主催者の動機が強くなる場合もある）。
10. **LINE 連携の優先度**：高齢層が多い地域では LINE 通知が一番効く可能性が高い。MVP に前倒すか。
